# 产品需求文档编写器 (PRD Writer)

帮助产品经理编写专业、规范、清晰的产品需求文档。基于提供的模板框架，灵活适配不同类型的产品需求。

## 何时使用

当用户有以下需求时立即触发：
- "写一份需求文档" / "帮我写PRD"
- "输出产品需求" / "写功能说明"
- "整理产品文档" / "写产品规格"
- 需要为某个功能编写详细的文档说明

## 核心原则

**模板是参考，不是约束**：
- 不必生搬硬套所有章节
- 根据实际需求灵活调整结构
- 以"需求能够清晰描述"为最高准则
- 可以增删章节，调整顺序

## 文档结构参考

### 1. 文档信息
```markdown
- **文档名称**：
- **产品名称**：
- **功能名称**：
- **版本**：
- **创建日期**：
- **创建人**：
```

### 2. 变更日志
| 版本号 | 修改日期 | 修改人 | 主要变更内容 | 备注 |
|--------|----------|--------|--------------|------|
| v1.0   |          |        | 初始版本     |      |

### 3. 产品/需求概述
- **需求背景**：为什么要做这个功能？当前痛点是什么？
- **需求价值**：解决什么问题？带来什么收益？
- **核心目标**：具体要达成什么目标？（可量化的更好）
- **名词解释**：关键术语、缩略词说明

### 4. 功能清单
列出本需求包含的所有功能点。

### 5. 关联模块
说明与哪些模块有关联，影响范围。

### 6. 核心业务流程
用流程图或文字描述主业务流程。

### 7. 全局说明
- **交互规则**：全局通用的交互规范
- **统一异常处理**：通用的错误处理方式

### 8. 功能详细描述
对每个功能模块进行详细描述：

#### 8.1 功能入口
用户如何进入这个功能。

#### 8.2 模块名称
- **功能说明**：这个模块做什么
- **前置条件**：使用这个功能需要什么前提
- **页面结构**：页面布局、元素组成
- **业务流程**：详细流程说明
- **业务规则&数据规则&字段说明**：
  - 业务规则：业务逻辑限制
  - 数据规则：数据验证规则
  - 字段说明：每个字段的含义、类型、限制
- **交互规则**：
  - 操作步骤1：触发条件 + 交互反馈
  - 操作步骤2：触发条件 + 交互反馈
  - ...
- **异常处理**：各种异常情况的处理方式
- **后置条件**：操作完成后的状态

### 9. 数据需求（可选）
- **埋点目的**：为什么要埋点
- **埋点事件**：需要埋哪些事件
- **核心指标**：关注哪些数据指标

## 写作指南

### DO（应该做的）
- **先沟通再写作**：写之前先和用户确认核心信息
- **具体清晰**：避免模糊的描述，如"很快"、"大量"
- **用户视角**：从用户角度描述功能和流程
- **可验证**：验收标准要可测试、可验证
- **图文并茂**：复杂流程建议配合流程图或示意图

### DON'T（避免的）
- **不要堆砌章节**：不需要的章节大胆删掉
- **不要复制模板**：把模板内容当示例，不要直接复制
- **不要遗漏关键信息**：背景、目标、流程、规则缺一不可
- **不要过度设计**：文档服务于需求理解，不是为了写而写

## 交互流程

1. **需求理解阶段**
   - 询问用户：产品类型、功能背景、目标用户
   - 确认文档的重点和必写章节
   - 了解是否有现成的资料（原型图、流程图等）

2. **信息收集阶段**
   - 功能清单确认
   - 核心流程梳理
   - 业务规则确认

3. **文档编写阶段**
   - 按照确认的章节结构编写
   - 先写框架，再填内容
   - 复杂部分与用户确认

4. **Review阶段**
   - 检查完整性
   - 检查清晰度
   - 根据反馈调整

## 示例片段

### 好的功能说明
```markdown
### 6.2 订单提交模块

**功能说明**
用户在购物车确认商品后，进入订单提交页面，填写收货信息、选择支付方式，完成订单创建。

**前置条件**
- 用户已登录
- 购物车中有商品
- 商品库存充足

**页面结构**
1. 顶部：返回按钮 + 页面标题"确认订单"
2. 收货信息区：显示默认收货地址，可点击更换
3. 商品列表区：展示商品图片、名称、规格、单价、数量
4. 配送方式：快递/自提选择
5. 优惠信息：可用优惠券列表
6. 底部：实付金额 + 提交订单按钮

**业务流程**
1. 用户点击"去结算"进入订单页
2. 系统加载默认收货地址和商品信息
3. 用户确认/修改收货信息
4. 用户选择配送方式和优惠券
5. 用户点击"提交订单"
6. 系统校验库存、地址完整性
7. 校验通过创建订单，跳转支付页

**业务规则&数据规则**
| 字段 | 规则 |
|------|------|
| 收货地址 | 必填，需包含省市区+详细地址+联系人+电话 |
| 商品库存 | 提交时实时校验，库存不足提示"商品库存不足" |
| 优惠券 | 仅可使用一张，需满足使用门槛 |

**交互规则**
- 操作1：点击收货地址 → 跳转地址列表页，可选中其他地址
- 操作2：点击优惠券 → 底部弹出优惠券选择面板
- 操作3：点击提交订单 → 按钮置灰显示"提交中..."，防止重复提交

**异常处理**
| 异常场景 | 处理方式 |
|----------|----------|
| 库存不足 | 弹窗提示"xx商品库存不足"，返回购物车 |
| 地址不完整 |  Toast提示"请完善收货地址" |
| 网络异常 | 提示"网络异常，请重试"，保留页面数据 |

**后置条件**
- 订单创建成功，状态为"待支付"
- 购物车中对应商品清除
- 优惠券标记为"已使用"
```

### 好的验收标准
```markdown
| 验收模块 | 验收项 | 验收标准 | 验证方式 |
|----------|--------|----------|----------|
| 订单创建 | 正常下单 | 填写完整信息后可成功创建订单，跳转支付页 | 手工测试 |
| 订单创建 | 库存校验 | 库存不足时阻止下单，给出明确提示 | 手工测试 |
| 订单创建 | 重复提交 | 快速点击提交按钮只创建一个订单 | 手工测试 |
| 地址选择 | 更换地址 | 可成功切换不同收货地址 | 手工测试 |
```

- [ ] 需求背景和目标清晰
- [ ] 功能清单完整
- [ ] 核心流程描述清楚
- [ ] 关键业务规则明确
- [ ] 异常场景覆盖全面
